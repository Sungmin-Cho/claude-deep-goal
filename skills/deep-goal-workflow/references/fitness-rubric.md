# fitness-rubric — goal 적합성 판정 기준

deep-goal의 2단계(적합성 평가)에서 사용자의 요청을 세 가지 판정 중 하나로 분류한다.

---

## 판정 기준 표

| 판정 | 신호 | 처리 |
|---|---|---|
| ✅ 적합 | 단일 목표 · 검증 가능한 종료조건 · "한 프롬프트보다 크고 백로그보다 작음" · 진행 증명 루프 존재 | 4~6단계 직행 |
| 🔧 재구성 | 종료조건 모호 / 범위 과대 / 증명 방법 부재 | 측정 가능화·분해·커맨드 식별 제안 |
| ⛔ 반려 | 검증 불가 주관 목표 · 단발성 · 무관한 잡다 목록 | 이유 + 대안 제시 |

---

## ✅ 적합 — 상세 신호

목표에 다음 신호가 모두(또는 대부분) 존재하면 적합하다:

1. **단일 목표**: 하나의 명확한 목적으로 수렴한다. "A도 하고 B도 하고 C도 하고 싶어"가 아니라 "A를 완수한다"의 형태.
2. **검증 가능한 종료조건**: 완료를 객관적으로 확인할 수 있다 (테스트 통과, 빌드 성공, 파일 존재, 빈 큐 등).
3. **적정 크기**: "한 프롬프트보다 크고 백로그보다 작음" — 한 턴으로 끝나지 않지만 무한히 늘어나는 로드맵도 아니다.
4. **진행 증명 루프 존재**: 목표를 향해 반복 진행이 가능하고, 각 단계에서 진전을 확인할 수 있다.

**적합 예시:**

- "이 저장소의 모든 TypeScript 컴파일 에러를 제거하라. `tsc --noEmit`가 에러 0으로 종료되면 완료." → 단일 목표, 검증 가능, 적정 크기, 이터레이션 가능
- "CLAUDE.md 작성 가이드라인에 맞춰 프로젝트 전체 문서를 정비하라. lint 경고 0 AND 리뷰 APPROVE가 종료조건." → 단일 목표, 검증 가능, 적정 크기

## 🔧 재구성 — 상세 신호 + 대응

다음 신호가 있으면 재구성 대화가 필요하다:

### 종료조건 모호
- **신호**: "더 좋게 만들어", "최대한 개선해", "어느 정도 완성되면 끝내도 돼"
- **대응**: 측정 가능한 기준으로 변환. "무엇이 달성되면 멈춰도 되는가? 어떤 커맨드/아티팩트로 확인하는가?"

### 범위 과대
- **신호**: 여러 독립적인 목표가 한 요청에 묶임. "전체 리팩터링 + 테스트 추가 + 문서화 + 배포까지"
- **대응**: 분해 제안. "이 목록 중 첫 번째 완수할 것 하나를 고르면, 그것만 goal로 만들겠다."

### 증명 방법 부재
- **신호**: 종료조건은 있지만 어떻게 확인할지 명시가 없다. "기능을 구현하면 끝."
- **대응**: 검증 커맨드 식별. "구현 완료를 어떻게 확인하는가? `npm test`, `make build`, 또는 특정 파일 존재?"

**재구성 예시:**

- "코드를 전반적으로 정리해줘" → 재구성: "어떤 파일 범위? lint 규칙 준수? 테스트 통과?" 명확화 후 적합으로 전환
- "이 프로젝트를 production-ready하게 만들어" → 재구성: production-ready의 체크리스트를 구체화, 범위를 한 sprint 단위로 분해

## ⛔ 반려 — 상세 신호 + 대안

구조적으로 goal에 적합하지 않은 경우 명확히 반려하고 대안을 제시한다.

### 검증 불가 주관 목표
- **신호**: "더 멋지게 만들어", "사용자가 좋아할 만하게", "전문가스럽게 느껴지도록"
- **이유**: Claude 평가자가 도구를 사용하지 않고 대화 출력만으로 종료를 판정하므로, 주관적 기준은 판정 불가
- **대안**: 주관 기준을 객관 지표로 바꿀 수 있으면 재구성으로 전환. 불가능하면 일반 대화 작업으로 처리

### 단발성
- **신호**: 한 턴이면 끝나는 요청. "이 함수 이름 바꿔줘", "package.json 버전 올려줘"
- **이유**: goal의 가치는 반복 진행에 있다. 단발은 오버헤드만 추가됨
- **대안**: 그냥 지금 바로 실행. goal 불필요

### 무관한 잡다 목록
- **신호**: "A도 하고 B도 하고 C도 D도 E도" — 서로 관련 없는 작업들의 목록
- **이유**: goal 공식 문서가 "무관한 잡다 목록은 goal에 적합하지 않다"고 경고
- **대안**: 목록에서 연관된 것들을 묶어 분리 goal 제안, 또는 `/loop`으로 순차 처리

**반려 예시:**

- "UI를 더 예쁘게 만들어줘" → 반려: 판정 기준 없음. 대안: "어떤 컴포넌트? 어떤 디자인 시스템 준수? 무엇이 통과되면 완료?" — 재구성 가능하면 전환
- "이메일 확인하고, 코드 리뷰하고, 문서 업데이트하고, 버그 2개 잡고, 테스트 추가해줘" → 반려: 무관한 잡다 목록. 대안: 버그 수정 + 테스트를 단일 goal로 분리, 나머지는 순서대로 개별 작업
